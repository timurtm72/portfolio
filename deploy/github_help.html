⚙️ Установка и настройка

Установка Git

sudo apt update && sudo apt install git
Устанавливает Git в Debian/Ubuntu

Проверка версии

git --version
Показывает установленную версию Git

Настройка имени пользователя

git config --global user.name "Ваше Имя"
Устанавливает имя для коммитов

Настройка email

git config --global user.email "your.email@example.com"
Устанавливает email для коммитов

Просмотр конфигурации

git config --list
Показывает все настройки Git

Редактор по умолчанию

git config --global core.editor nano
Устанавливает nano как редактор по умолчанию

Цветной вывод

git config --global color.ui auto
Включает цветной вывод команд

Настройка алиасов

git config --global alias.st status
Создаёт алиас: git st вместо git status
📖 Пример: Полная первоначальная настройка 
# Установка Git
sudo apt update && sudo apt install git

# Настройка пользователя
git config --global user.name "Иван Иванов"
git config --global user.email "ivan@example.com"

# Настройка редактора
git config --global core.editor nano

# Включение цвета
git config --global color.ui auto

# Полезные алиасы
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.ci commit
git config --global alias.st status
git config --global alias.unstage 'reset HEAD --'
git config --global alias.last 'log -1 HEAD'

# Проверка настроек
git config --list

Настройка SSH ключа для GitHub

ssh-keygen -t ed25519 -C "your.email@example.com"
Генерирует SSH ключ

Копирование публичного ключа

cat ~/.ssh/id_ed25519.pub
Показывает публичный ключ для добавления на GitHub

Тест SSH соединения с GitHub

ssh -T git@github.com
Проверяет подключение к GitHub
💡 Уровни конфигурации Git:
  • --system - для всех пользователей системы (/etc/gitconfig)
  • --global - для текущего пользователя (~/.gitconfig)
  • --local - для конкретного репозитория (.git/config)

📚 Основы Git

Создание нового репозитория

git init
Инициализирует Git репозиторий в текущей директории

Клонирование репозитория

git clone https://github.com/user/repo.git
Клонирует удаленный репозиторий

Клонирование в конкретную папку

git clone https://github.com/user/repo.git my-folder
Клонирует репозиторий в папку my-folder

Статус репозитория

git status
Показывает состояние файлов (изменены, добавлены, удалены)

Краткий статус

git status -s
Показывает статус в сокращённом виде

Добавление файла в staging

git add файл.txt
Добавляет файл в индекс для коммита

Добавление всех файлов

git add .
Добавляет все изменённые файлы

Добавление по маске

git add *.js
Добавляет все файлы .js

Интерактивное добавление

git add -p
Позволяет выбрать части файла для добавления

Удаление файла

git rm файл.txt
Удаляет файл и добавляет это изменение в staging

Удаление из Git, но не из файловой системы

git rm --cached файл.txt
Удаляет файл из Git, но оставляет в папке

Переименование файла

git mv старое-имя.txt новое-имя.txt
Переименовывает файл в Git
📖 Пример: Создание первого репозитория 
# Создаём директорию проекта
mkdir my-project
cd my-project

# Инициализируем Git
git init

# Создаём файл
echo "# My Project" > README.md

# Добавляем в staging
git add README.md

# Создаём первый коммит
git commit -m "Initial commit"

# Проверяем статус
git status
# Должно быть: nothing to commit, working tree clean

Просмотр изменений

git diff
Показывает изменения в файлах (не в staging)

Изменения в staging

git diff --staged
Показывает изменения, добавленные в staging

Игнорирование файлов (.gitignore)

nano .gitignore
Создаёт/редактирует файл .gitignore
📖 Пример: Содержимое .gitignore

# Операционная система
.DS_Store

# Временные файлы

# Скомпилированные файлы
*.o

💾 Коммиты

Создание коммита

git commit -m "Описание изменений"
Создаёт коммит с сообщением

Многострочное сообщение

git commit
Открывает редактор для детального сообщения

Коммит с добавлением всех изменений

git commit -am "Сообщение"
Добавляет все изменённые файлы и создаёт коммит

Изменение последнего коммита

git commit --amend -m "Новое сообщение"
Изменяет сообщение последнего коммита

Добавление файлов в последний коммит

git add забытый-файл.txt && git commit --amend --no-edit
Добавляет файл в последний коммит без изменения сообщения

Просмотр истории коммитов

git log
Показывает историю коммитов

Краткая история

git log --oneline
Показывает по одному коммиту на строку

История с графиком

git log --graph --oneline --all
Показывает графическое представление веток

Последние N коммитов

git log -5
Показывает последние 5 коммитов

История конкретного файла

git log -- файл.txt
Показывает коммиты, изменившие файл

Поиск в коммитах

git log --grep="исправление"
Ищет коммиты по тексту сообщения

Коммиты конкретного автора

git log --author="Имя Автора"
Показывает коммиты определённого автора

Изменения в коммите

git show HEAD
Показывает последний коммит с изменениями

Конкретный коммит

git show abc1234
Показывает коммит по хешу
💡 Правила хороших коммитов:
  • Используйте повелительное наклонение: "Добавить функцию", а не "Добавлена функция"
  • Первая строка до 50 символов: краткое описание
  • Пустая строка после первой: затем детальное описание
  • Атомарность: один коммит = одна логическая единица изменений
  • Осмысленность: "Исправить баг" → "Исправить ошибку валидации email"
📖 Пример: Структура хорошего коммита 
# Краткое описание (до 50 символов)
Добавить валидацию формы регистрации

# Пустая строка

# Детальное описание (при необходимости)
- Добавлена проверка email
- Добавлена проверка сложности пароля
- Минимальная длина пароля 8 символов
- Обязательное наличие цифр и букв

# Связанные issues (если есть)
Fixes #123

🌿 Ветки (Branches)

Список веток

git branch
Показывает все локальные ветки

Все ветки (включая удалённые)

git branch -a
Показывает локальные и удалённые ветки

Создание новой ветки

git branch feature-login
Создаёт новую ветку feature-login

Переключение на ветку

git checkout feature-login
Переключается на ветку feature-login

Создание и переключение (одной командой)

git checkout -b feature-payment
Создаёт ветку feature-payment и переключается на неё

Новый способ (с Git 2.23+)

git switch feature-login
Переключается на ветку (новая команда)

Создание и переключение (новый способ)

git switch -c feature-payment
Создаёт и переключается на ветку

Переименование ветки

git branch -m старое-имя новое-имя
Переименовывает ветку

Переименование текущей ветки

git branch -m новое-имя
Переименовывает текущую ветку

Удаление ветки

git branch -d feature-login
Удаляет ветку (безопасно, только если слита)

Принудительное удаление

git branch -D feature-login
Удаляет ветку принудительно

Слияние ветки

git merge feature-login
Сливает ветку feature-login в текущую ветку

Слияние без fast-forward

git merge --no-ff feature-login
Создаёт merge commit даже при возможности fast-forward

Отмена слияния

git merge --abort
Отменяет слияние при конфликтах
📖 Пример: Типичный workflow с ветками 
# Находимся в ветке main
git checkout main

# Создаём и переключаемся на новую ветку для функции
git checkout -b feature-user-profile

# Работаем, делаем изменения
echo "Profile page" > profile.html
git add profile.html
git commit -m "Добавить страницу профиля"

# Ещё работаем
echo "Settings" > settings.html
git add settings.html
git commit -m "Добавить страницу настроек"

# Возвращаемся в main
git checkout main

# Сливаем изменения
git merge feature-user-profile

# Удаляем ветку (она больше не нужна)
git branch -d feature-user-profile
📖 Пример: Разрешение конфликтов 
# Пытаемся слить ветку
git merge feature-branch
# CONFLICT (content): Merge conflict in file.txt

# Смотрим конфликтующие файлы
git status

# Редактируем файл, разрешаем конфликт
# Убираем маркеры <<<<<<< ======= >>>>>>>
nano file.txt

# Добавляем разрешённые файлы
git add file.txt

# Завершаем слияние
git commit -m "Слить feature-branch, разрешить конфликты"

# Или отменяем слияние
git merge --abort

Rebase (перебазирование)

git rebase main
Перемещает коммиты текущей ветки на вершину main

Интерактивный rebase

git rebase -i HEAD~3
Позволяет редактировать последние 3 коммита

Отмена rebase

git rebase --abort
Отменяет rebase при проблемах
⚠️ Важно: Никогда не делайте rebase публичных веток! Rebase переписывает историю, что может создать проблемы для других разработчиков. Используйте rebase только для локальных веток или до push.

☁️ Удалённые репозитории

Список удалённых репозиториев

git remote -v
Показывает все удалённые репозитории с URL

Добавление удалённого репозитория

git remote add origin https://github.com/user/repo.git
Добавляет удалённый репозиторий с именем origin

Изменение URL удалённого репозитория

git remote set-url origin https://github.com/user/new-repo.git
Изменяет URL удалённого репозитория

Удаление удалённого репозитория

git remote remove origin
Удаляет удалённый репозиторий

Информация об удалённом репозитории

git remote show origin
Показывает детальную информацию об origin

Получение изменений (fetch)

git fetch origin
Загружает изменения, но не сливает их

Получение и слияние (pull)

git pull origin main
Загружает и сливает изменения из ветки main

Pull с rebase

git pull --rebase origin main
Загружает изменения и делает rebase вместо merge

Отправка изменений (push)

git push origin main
Отправляет коммиты в удалённую ветку main

Отправка новой ветки

git push -u origin feature-branch
Отправляет новую ветку и устанавливает отслеживание

Принудительная отправка

git push --force origin main
Принудительно перезаписывает удалённую ветку

Безопасная принудительная отправка

git push --force-with-lease origin main
Безопаснее --force, проверяет изменения других

Удаление удалённой ветки

git push origin --delete feature-branch
Удаляет ветку на удалённом сервере
📖 Пример: Связывание локального репозитория с GitHub 
# Создаём локальный репозиторий
git init
echo "# My Project" > README.md
git add README.md
git commit -m "Initial commit"

# Добавляем удалённый репозиторий
git remote add origin https://github.com/username/repo.git

# Отправляем изменения
git push -u origin main

# Проверяем подключение
git remote -v
📖 Пример: Работа с форком 
# Клонируем свой форк
git clone https://github.com/your-username/repo.git
cd repo

# Добавляем оригинальный репозиторий как upstream
git remote add upstream https://github.com/original-owner/repo.git

# Получаем изменения из upstream
git fetch upstream

# Сливаем изменения из upstream/main в свой main
git checkout main
git merge upstream/main

# Отправляем обновления в свой форк
git push origin main
💡 Разница между fetch и pull:
  • git fetch - загружает изменения, но НЕ сливает их. Безопасно для просмотра изменений
  • git pull - загружает И сливает изменения. Эквивалентно: git fetch + git merge
  • git pull --rebase - загружает и делает rebase. Создаёт более чистую историю

🐙 Работа с GitHub

Создание репозитория на GitHub

1. Перейдите на github.com
2. Нажмите "New repository"
3. Заполните название и описание
4. Выберите публичный или приватный
5. Нажмите "Create repository"

Клонирование через HTTPS

git clone https://github.com/username/repo.git
Клонирует репозиторий через HTTPS

Клонирование через SSH

git clone git@github.com:username/repo.git
Клонирует через SSH (требуется настроенный ключ)

Fork репозитория

1. Откройте репозиторий на GitHub
2. Нажмите кнопку "Fork" вверху справа
3. Выберите свой аккаунт
4. GitHub создаст копию репозитория

Создание Pull Request

1. Отправьте изменения в свою ветку: git push origin feature-branch
2. На GitHub откройте свой репозиторий
3. Нажмите "Compare & pull request"
4. Заполните описание изменений
5. Нажмите "Create pull request"

Синхронизация форка

git fetch upstream && git merge upstream/main
Обновляет форк из оригинального репозитория
📖 Пример: Вклад в open-source проект 
# 1. Форкаем репозиторий через веб-интерфейс GitHub

# 2. Клонируем свой форк
git clone https://github.com/your-username/project.git
cd project

# 3. Добавляем upstream
git remote add upstream https://github.com/original-owner/project.git

# 4. Создаём ветку для изменений
git checkout -b fix-typo

# 5. Делаем изменения
echo "Fixed typo" > file.txt
git add file.txt
git commit -m "Исправить опечатку в file.txt"

# 6. Отправляем в свой форк
git push origin fix-typo

# 7. Создаём Pull Request через веб-интерфейс

# 8. После мерджа синхронизируем форк
git checkout main
git fetch upstream
git merge upstream/main
git push origin main

# 9. Удаляем ветку
git branch -d fix-typo
git push origin --delete fix-typo

GitHub Issues

Issues - это система отслеживания задач на GitHub:
- Баги
- Новые функции
- Вопросы
- Обсуждения

Связь коммита с Issue

git commit -m "Исправить баг со входом, fixes #42"
Автоматически закрывает Issue #42 при мердже
💡 Ключевые слова для закрытия Issues:
  • close, closes, closed - закрывает issue
  • fix, fixes, fixed - закрывает issue
  • resolve, resolves, resolved - закрывает issue
Пример: "Fixes #123" в сообщении коммита автоматически закроет Issue #123

GitHub Actions (базовый пример)

GitHub Actions - это CI/CD платформа для автоматизации
📖 Пример: Простой GitHub Actions workflow 
# Создаём файл .github/workflows/test.yml

name: Tests

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    
    steps:
    - uses: actions/checkout@v2
    
    - name: Set up Node.js
      uses: actions/setup-node@v2
      with:
        node-version: '16'
    
    - name: Install dependencies
      run: npm install
    
    - name: Run tests
      run: npm test

Защищённые ветки

Settings → Branches → Add rule:
- Требовать review перед мерджем
- Требовать успешные тесты
- Запретить force push
- Требовать линейную историю

👥 Совместная работа

Просмотр авторов файла

git blame файл.txt
Показывает, кто и когда изменял каждую строку

Сокращённый формат blame

git blame -L 10,20 файл.txt
Показывает информацию только для строк 10-20

Временное сохранение изменений (stash)

git stash
Сохраняет текущие изменения в стек

Stash с сообщением

git stash save "WIP: работа над функцией"
Сохраняет stash с описанием

Список stash

git stash list
Показывает все сохранённые stash

Применение stash

git stash apply
Применяет последний stash (не удаляет его)

Применение и удаление stash

git stash pop
Применяет и удаляет последний stash

Удаление stash

git stash drop
Удаляет последний stash
📖 Пример: Использование stash 
# Работаем над функцией
echo "половина работы" > feature.js
git add feature.js

# Внезапно нужно срочно исправить баг в main
# Сохраняем текущую работу
git stash save "WIP: новая функция"

# Переключаемся на main
git checkout main

# Исправляем баг
echo "исправление" > fix.js
git add fix.js
git commit -m "Исправить критический баг"

# Возвращаемся к работе над функцией
git checkout feature-branch
git stash pop

# Продолжаем работу
echo "завершение работы" >> feature.js
git add feature.js
git commit -m "Завершить новую функцию"

Cherry-pick (применение конкретного коммита)

git cherry-pick abc1234
Применяет изменения из коммита abc1234 в текущую ветку

Cherry-pick без коммита

git cherry-pick --no-commit abc1234
Применяет изменения, но не создаёт коммит

Просмотр изменений между ветками

git diff main..feature-branch
Показывает различия между ветками

Коммиты в одной ветке, но не в другой

git log main..feature-branch
Показывает коммиты только в feature-branch

Очистка удалённых веток

git remote prune origin
Удаляет ссылки на удалённые ветки, которых больше нет на сервере
📖 Пример: Workflow команды 
# Разработчик A: Создаёт функцию
git checkout -b feature-auth
# ... работа ...
git commit -m "Добавить аутентификацию"
git push origin feature-auth
# Создаёт Pull Request

# Разработчик B: Код ревью
# Просматривает изменения на GitHub
# Оставляет комментарии

# Разработчик A: Вносит правки
git checkout feature-auth
# ... исправления ...
git commit -m "Исправить замечания из ревью"
git push origin feature-auth

# Разработчик B: Одобряет и мерджит
# Нажимает "Merge pull request" на GitHub

# Разработчик A: Синхронизируется
git checkout main
git pull origin main
git branch -d feature-auth

📜 История и откат изменений

Отмена изменений в файле

git restore файл.txt
Отменяет изменения в файле (не в staging)

Старый способ отмены

git checkout -- файл.txt
Отменяет изменения (устаревший способ)

Убрать файл из staging

git restore --staged файл.txt
Убирает файл из staging, но сохраняет изменения

Старый способ убрать из staging

git reset HEAD файл.txt
Убирает из staging (устаревший способ)

Откат последнего коммита (сохранить изменения)

git reset --soft HEAD~1
Отменяет коммит, но сохраняет изменения в staging

Откат последнего коммита (убрать из staging)

git reset --mixed HEAD~1
Отменяет коммит, изменения остаются, но не в staging

Откат последнего коммита (удалить изменения)

git reset --hard HEAD~1
Полностью удаляет коммит и все изменения

Откат к конкретному коммиту

git reset --hard abc1234
Откатывается к коммиту abc1234, удаляя всё после него

Безопасный откат (создаёт новый коммит)

git revert HEAD
Создаёт новый коммит, отменяющий изменения последнего

Revert конкретного коммита

git revert abc1234
Создаёт коммит, отменяющий изменения из abc1234
⚠️ Важно понимать разницу:
  • git reset - изменяет историю (опасно для публичных веток!)
  • git revert - создаёт новый коммит отмены (безопасно для публичных веток)
Используйте revert для публичных веток, reset только для локальных!
📖 Пример: Откат изменений (разные сценарии) 
# Сценарий 1: Отменить изменения в файле (не добавлен в staging)
echo "ошибка" > file.txt
git restore file.txt  # Файл вернётся к последнему коммиту

# Сценарий 2: Убрать файл из staging
echo "изменение" > file.txt
git add file.txt
git restore --staged file.txt  # Убрано из staging, изменения остались

# Сценарий 3: Исправить последний коммит
git commit -m "Добавить фичу"
# Ой, забыли файл!
git add forgotten-file.txt
git commit --amend --no-edit

# Сценарий 4: Откатить последний коммит (локально)
git reset --soft HEAD~1  # Коммит отменён, файлы в staging
# Вносим правки
git commit -m "Исправленная версия"

# Сценарий 5: Отменить коммит (уже в origin)
git revert HEAD  # Создаёт новый коммит отмены
git push origin main  # Безопасно отправляем

Поиск потерянных коммитов (reflog)

Восстановление после reset

🔄 Рабочие процессы (Workflows)

Git Flow

Классический workflow с несколькими постоянными ветками
💡 Структура Git Flow:
  • main - стабильная продакшен версия
  • develop - основная ветка разработки
  • feature/* - ветки для новых функций
  • release/* - подготовка к релизу
  • hotfix/* - срочные исправления продакшена
📖 Пример: Git Flow workflow 
# Инициализация
git checkout -b develop

# Создание feature
git checkout -b feature/user-auth develop
# ... работа ...
git add .
git commit -m "Добавить аутентификацию"

# Завершение feature
git checkout develop
git merge --no-ff feature/user-auth
git branch -d feature/user-auth
git push origin develop

# Создание release
git checkout -b release/1.0.0 develop
# ... финальные правки ...
git commit -am "Bump version to 1.0.0"

# Завершение release
git checkout main
git merge --no-ff release/1.0.0
git tag -a v1.0.0 -m "Release version 1.0.0"
git checkout develop
git merge --no-ff release/1.0.0
git branch -d release/1.0.0

# Hotfix
git checkout -b hotfix/1.0.1 main
# ... исправление ...
git commit -am "Исправить критический баг"
git checkout main
git merge --no-ff hotfix/1.0.1
git tag -a v1.0.1 -m "Hotfix 1.0.1"
git checkout develop
git merge --no-ff hotfix/1.0.1
git branch -d hotfix/1.0.1

GitHub Flow

Упрощённый workflow для быстрой разработки
💡 Принципы GitHub Flow:
  • Одна main ветка, всегда готовая к деплою
  • Все фичи в отдельных ветках
  • Pull Request для каждого изменения
  • Code review обязателен
  • Деплой после мерджа в main
📖 Пример: GitHub Flow workflow 
# Создаём ветку для функции
git checkout -b add-search-feature main

# Разрабатываем
# ... работа ...
git add .
git commit -m "Реализовать поиск"
git push origin add-search-feature

# Создаём Pull Request на GitHub
gh pr create --title "Добавить функцию поиска"

# После ревью и одобрения - мерджим на GitHub

# Локально обновляемся
git checkout main
git pull origin main
git branch -d add-search-feature

# Деплоим
# ... процесс деплоя ...

Trunk Based Development

Минималистичный подход - все коммитят в main
💡 Принципы Trunk Based:
  • Все работают в одной ветке (main/trunk)
  • Очень короткоживущие ветки (часы, не дни)
  • Частые коммиты и интеграции
  • Feature flags для неготовых функций
  • Сильный акцент на CI/CD и тестирование
📖 Пример: Сравнение workflows
Аспект Git Flow GitHub Flow Trunk Based
Сложность Высокая Средняя Низкая
Основные ветки main, develop main main
Жизнь feature-веток Недели Дни Часы
Подходит для Релизы с версиями Continuous Deployment Высокая скорость
Размер команды Средние/большие Любой Опытные команды
📖 Пример: Semantic Versioning в Git 
# Создание тега версии
git tag -a v1.2.3 -m "Release version 1.2.3"

# Отправка тега на GitHub
git push origin v1.2.3

# Отправка всех тегов
git push origin --tags

# Список тегов
git tag -l

# Checkout конкретной версии
git checkout v1.2.3

# Удаление тега
git tag -d v1.2.3
git push origin --delete v1.2.3

# Semantic Versioning: MAJOR.MINOR.PATCH
# 1.0.0 - Major: несовместимые изменения API
# 1.1.0 - Minor: новая функциональность (совместима)
# 1.1.1 - Patch: исправления багов
